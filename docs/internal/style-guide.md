# Style Guide

## Language And Chapter Shape

- เขียนภาษาไทยตรง ชัด และสมมติว่าผู้เรียนรู้ TypeScript แต่ยังไม่รู้ Angular
- อธิบาย Angular mental model ก่อน decorator/function/operator syntax
- บทลงมือทำมี goal, prerequisites, changed files, code steps, expected UI/API results, common errors, exercise and checkpoint
- code block ควรไม่เกินประมาณ 30 บรรทัดถ้าแบ่งได้
- browser lesson ระบุ viewport, interaction, expected state และ console/network checks

## Angular Rules

- Angular 22 is the manuscript baseline; examples must not silently fall back to older Angular APIs
- standalone components and functional providers are the primary architecture
- use the Angular 2025 concise file naming generated by CLI 22, such as `app.ts`, unless a chapter explicitly explains existing naming
- teach zoneless rendering through explicit Angular notifications such as Signals, template listeners and `AsyncPipe`; do not rely on Zone.js as the mental model
- component inputs are read-only contracts; changes go upward through explicit outputs
- synchronous local state uses Signals; derived values use `computed`
- do not use `effect` to copy or derive state that `computed` can express
- services model behavior/dependencies, not a global dumping ground
- new root services use `@Service()` and `inject()`; use `@Injectable` only when constructor DI, non-root `providedIn` or advanced provider configuration requires it
- choose provider scope intentionally and explain instance lifetime
- Observable streams use AsyncPipe or an explicit interop boundary; manual subscriptions require ownership and cleanup rationale
- use `switchMap` when a newer route/filter value should cancel stale work
- use stable Signal Forms from `@angular/forms/signals` as the canonical form implementation
- explain Reactive Forms as existing-code/migration context without maintaining a second Product Form implementation
- URL state belongs to Router query parameters; HTTP state is not duplicated into a global writable store
- functional interceptors handle cross-cutting HTTP concerns, not page-specific errors
- lazy load feature routes only after the basic Router mental model is understood

## TypeScript Rules

- strict mode; no `any`, unsafe double assertions or unexplained non-null assertions
- define explicit input/output/request/response/problem types
- external JSON starts as `unknown` and passes runtime guards
- use discriminated unions for async/form states when they prevent impossible combinations
- type-only imports use `import type` where appropriate

## Naming

- Project: `product-catalog-angular`
- Angular classes/types: PascalCase; files/folders/selectors: kebab-case
- Components: `ProductCard`, `ProductForm`, `ProductListPage`
- Services: `ProductApi`; use Angular 2025 concise names such as `product-api.ts`, not `product-api.service.ts`
- Signals: noun names such as `searchTerm`; private writable state uses clear ownership, public exposure is read-only
- Routes: `/products`, `/products/new`, `/products/:productId`, `/products/:productId/edit`
- Environment values: `apiBaseUrl`, `useMockApi`; never store secrets

## Routing And API

- use links for navigation and buttons for commands
- parse/narrow route and query parameters before use
- use runtime URL from Angular CLI; do not fix ports
- mock and real API share routes, statuses and body shapes
- do not show cancelled requests as errors or expose unsafe server details
- mock startup must be explicit and development-only

## Testing

- use the Angular CLI Vitest setup; do not teach a parallel Karma path
- test user-visible behavior, not private fields, lifecycle call counts or implementation details
- configure TestBed with the smallest meaningful provider/import boundary
- use `provideHttpClientTesting` and `HttpTestingController` for focused HTTP tests
- verify outstanding HTTP requests after each relevant test
- test loading, empty, error, validation, conflict and recovery, not only happy path
